# PR 2.2 — Process Tracing

**Epoch:** 2 (Core Intelligence)
**Depends on:** PR 2.1 (Call Resolution), PR 1.3 (Advanced Traversal)
**Estimated effort:** 2-3 weeks
**Impact:** CRITICAL — feeds Battle Testing + Compliance Validator

## Context

With call resolution (PR 2.1), we know what calls what. Process tracing builds on this to answer "what is the full execution flow from entry point to leaf?" and "what changed in the codebase and what processes are affected?"

This directly enables Orquestra's Compliance Validator (pre-commit analysis of affected processes) and Battle Testing L3-L5 (process trace verification across agent harnesses).

## Deliverables

### 1. Entry Point Detection Plugin
- New file: `graphify/entry_points.py`
  - `EntryPointDetector` protocol — `detect(graph) → List[EntryPoint]`
  - `EntryPoint` dataclass: `name`, `kind` (HTTP, CLI, CRON, TEST, EVENT, EXPORT), `file`, `line`, `language`, `framework`
  - `FrameworkEntryPointDetector` — scans graph for framework-specific patterns:
    - Next.js: page.tsx, route.ts, middleware.ts files
    - Express: `app.get()`, `app.post()`, `router.verb()` calls
    - Flask: `@app.route()` decorators
    - FastAPI: `@app.get()`, `@app.post()` decorators
    - Django: urls.py `urlpatterns`, views
    - CLI: `def main()`, `if __name__ == "__main__"`, `func main()`
    - Go: `func main()`, `func init()`
    - Test fixtures: `def test_*`, `it(`, `describe(`, `@Test`
    - Cron: `@app.cron`, `node-cron`, `time.Ticker`
    - Events: `EventEmitter.on()`, signal handlers, `addEventListener`
  - `register_detector(framework_name, detector)` — plugin registry, same pattern as `LanguageCallProvider`
  - `score_entry_points(entry_points, graph) → List[(EntryPoint, score)]` — heuristics:
    - Route handler > CLI > test > library export
    - Entry call chain depth (deeper = more important)
    - Number of direct+transitive dependents

### 2. Process Tracing Engine
- New file: `graphify/processes.py`
  - `trace_process(entry_point, graph, max_depth=20, max_nodes=1000) → Process`
  - BFS along `CALLS` edges from entry point, deduplicating cycles (visited set)
  - `Process` dataclass:
    - `name` — e.g., "POST /api/auth/login"
    - `entry_point` — the `EntryPoint`
    - `steps` — `List[ProcessStep]` where `ProcessStep = {node_id, label, file, line, depth, callers: [], callees: []}`
    - `total_steps`, `max_depth`, `language`, `framework`
    - `confidence` — average of all `CALLS` edge confidences in the trace
    - `cyclomatic_complexity` — McCabe complexity sum across all nodes in trace
    - `external_touches` — count of calls to external libraries/frameworks
  - `trace_all_entry_points(graph) → List[Process]` — trace every detected entry point
  - `trace_changed_nodes(graph, changed_files) → List[Process]` — only trace processes affected by changed files
  - Uses PR 1.3's bidirectional BFS for fast shortest-path tracing

### 3. Process Clustering
- New file: `graphify/process_cluster.py`
  - `cluster_processes(processes, overlap_threshold=0.9) → List[ProcessCluster]`
  - Groups processes by shared call paths: if two traces share >90% symbols, they're variants of the same process
  - `deduplicate(processes) → List[Process]` — keep the most detailed (deepest) variant, discard near-duplicates
  - `merge_cluster(cluster) → Process` — produces canonical process trace for a cluster
  - Output: `ProcessCluster` with `canonical: Process`, `variants: List[Process]`, `cohesion_score`

### 4. Change Detection
- New file: `graphify/change_detect.py`
  - `detect_changes(graph, git_diff_text=None, scope="all") → ChangeReport`
  - If `git_diff_text` provided, parse diff for changed files + lines
  - If not, run `git diff HEAD~1` in repo root
  - `find_affected_symbols(graph, changed_files, changed_lines) → List[AffectedSymbol]`
    - Match changed lines to `SourceLocation` ranges on nodes
    - Direct: node's source location overlaps changed region
    - Indirect: node is called by a directly affected node (1-hop)
  - `find_affected_processes(graph, affected_symbols) → List[AffectedProcess]`
    - Which processes contain any affected symbol in their trace?
    - `affected_steps` — which steps in the process are affected
  - `assess_risk(affected_count, affected_process_count, max_depth) → risk_level`
    - >50 affected + >3 processes + depth >5 → CRITICAL
    - >20 affected + >1 process → HIGH
    - >5 affected → MEDIUM
    - else → LOW
  - `generate_recommendations(change_report) → List[str]`
    - "Test process X (affected steps: Y, Z)"
    - "Review blast radius: N symbols at depth D"
    - "Consider integration tests for processes: A, B, C"

### 5. Process Persistence
- Modified: `graphify/build.py`
  - After call resolution, run process tracing and write `STEP_IN_PROCESS` edges
  - Write `graphify-out/processes.json` — all process traces in structured format
  - `G.graph["process_stats"] = {traced: 14, total_steps: 847, clustered: 9}`
- New CLI subcommand: `graphify processes`
  - `graphify processes list` — list all detected entry points with scores
  - `graphify processes trace --entry-point "POST /api/auth/login"` — trace single process
  - `graphify processes detect-changes` — run change detection on current uncommitted diff

### 6. MCP Tools
- New: `detect_changes` tool in `serve.py`
  ```json
  detect_changes({scope: "all"})
    → summary: {changed_count, affected_count, risk_level, changed_files: [...]}
    → changed_symbols: [{name, kind, file, changed_lines: [start, end]}]
    → affected_processes: [{name, step_count, affected_steps: [1, 3, 5], affected_nodes: [...]}]
    → recommendations: [{action, reason}]
  ```
- New: `trace` tool in `serve.py`
  ```json
  trace({entry_point: "POST /api/auth/login"})
    → process: {name, entry_point, total_steps, max_depth, confidence, cyclomatic_complexity}
    → steps: [{index, node_id, label, file, line, depth, callers: [...], callees: [...]}]
  ```
- Updated: `context` tool (from PR 2.1) now includes process membership:
  ```json
  context({name: "validateUser"})
    → processes: [{name, step_index, total_steps, entry_point}]
  ```
