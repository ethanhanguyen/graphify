# PR 4.2 — Ranked Node Lookup

## Problem
`_find_node(G, "uri.ts")` returns `extensions/terminal-suggest/src/helpers/uri.ts`
instead of `src/vs/base/common/uri.ts`. First substring match wins.
No ranking, exact-match preference, or file path depth scoring.

## Goal
`_find_node` returns the most relevant node for ambiguous labels.

## Scoring Design
For each candidate node matching the search term:
1. **Exact label match**: +100 — highest priority
2. **Source file name match**: +50 — label matches filename
3. **Path depth**: +max(0, 30 - slashes) — shallow = core module
4. **Degree**: +min(degree, 20) — central nodes rank higher
5. Break ties by shortest node ID

## Changes

### 1. `graphify/serve.py` — Ranked lookup
Rewrite `_find_node` with scoring, keep backward-compatible signature.

### 2. `graphify/__main__.py` — Path CLI source/target matching
`path` command currently uses `_score_nodes` for source/target.
Switch to `_find_node` for accurate single-node resolution.

### 3. `tests/test_serve.py` — Ranked lookup tests
- Exact match beats substring match
- Shorter file path beats deeper path
- Higher degree beats lower degree (same label depth)
- No match returns empty list
- Diacritic-insensitive matching preserved
