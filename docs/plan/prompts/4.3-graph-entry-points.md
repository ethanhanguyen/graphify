# PR 4.3 — Graph-Based Entry Point Detection

## Problem
`graphify processes <path> list` always returns 0 results.
`detect_entry_points()` requires extraction dicts (raw AST/semantic data)
that are discarded after build. The graph.json has no extraction data.

## Goal
Detect entry points from graph node attributes alone. No extraction dicts needed.

## Detection Strategies

### A. Label/File pattern matching (label + source_file + node_type)
- **CLI/Entry**: `main`, `run`, `start`, `index`, `init`, `bootstrap`, `server`, `app`
  - node_type must be FUNCTION/METHOD/CLASS
- **HTTP handler**: source_file contains `route`, `controller`, `handler`, `api`, `endpoint`
  - node_type must be FUNCTION/METHOD
- **Tests**: label matches `test_*`, `Test*`, `*Test`, `*_test`

### B. Graph structure analysis
- Nodes with high in-degree CALLS edges + zero outgoing CALLS = top-level entry
- Degree > 3 and no CALLS into it from others in same file → likely entry point

## Changes

### 1. `graphify/entry_points.py` — GraphEntryPointDetector
Add `GraphEntryPointDetector` class with two-stage detection (pattern + structure).

### 2. `graphify/entry_points.py` — Fallback in detect_entry_points()
When `extractions` produces no results, invoke graph-based detection.

### 3. `graphify/__main__.py` — Fix processes list
Remove per-language retry loop. Use unified detection that falls back to graph.

### 4. `tests/test_entry_points.py` — Graph detection tests
- Pattern matching: main(), test_foo, route handler
- Structure-based: high-in-degree + no-outgoing-calls
- Empty graph → no entry points
- Integration: processes list CLI with graph-only data
