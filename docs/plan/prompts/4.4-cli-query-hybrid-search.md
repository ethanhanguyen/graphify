# PR 4.4 — CLI Query via Hybrid Search

## Problem
`graphify query` uses O(N) keyword substring scoring + shallow BFS.
All 15 VSCode benchmark queries return the same output nodes.
The BM25 search engine exists in `graphify/search/` but is only
used by the MCP server, not the CLI.

## Goal
CLI `query` produces relevant, diverse results using BM25.

## Changes

### 1. `graphify/search/__init__.py` — CLI-friendly builder
Expose `build_orchestrator(graph, use_embeddings=False)`.
Ensure `SearchOrchestrator` works without SentenceTransformer.

### 2. `graphify/__main__.py` — Query rewrite
Replace naive scoring (lines 1227-1234) with BM25 search.
Keep existing fallback for environments where search fails.

BM25 mode:
```
orch = build_search_orchestrator(G, use_embeddings=False)
result = orch.search(question, {"limit": 15})
nodes = {nid for nid, _ in result["results"]}
edges = collect_edges(G, nodes)
render subgraph to text as before
```

### 3. Edge collection helper
Collect all edges incident between result nodes for `_subgraph_to_text`.

### 4. `tests/test_search_orchestrator.py` — Additional tests
- `build_orchestrator` without embeddings
- BM25 search on known graph
- Graceful degradation when search import fails

## Verification
```bash
python -m pytest tests/test_search_orchestrator.py tests/test_search_bm25.py -q --tb=short
python -m graphify query "how does lifecycle work" --graph <fixture>  # diverse results
```
